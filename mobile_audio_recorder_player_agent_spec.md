# 모바일 녹음·재생 웹앱 개발 지시서

## 1. 프로젝트 목적

스마트폰에서 사용자가 직접 노래를 녹음하고, 저장된 녹음 파일을 일반 음악 플레이어처럼 재생할 수 있는 모바일 웹앱을 개발한다.

이 앱은 정식 앱스토어 배포 없이 HTTPS 웹사이트로 배포하며, 우선 Android 스마트폰에서 개발·테스트한다. 그러나 구조와 사용 API는 Android 전용으로 만들지 말고, iPhone의 Safari 및 Brave에서도 사용할 수 있도록 웹 표준과 WebKit의 제약을 기준으로 설계한다.

주요 사용자는 한 명 또는 소수이며, 로그인과 서버 동기화 없이 스마트폰 내부에 녹음 파일을 저장하는 **로컬 우선(local-first) 웹앱**으로 시작한다.

---

## 2. 가장 중요한 개발 원칙

1. Android에서만 작동하는 기능이나 Chromium 전용 API에 의존하지 않는다.
2. iPhone의 모든 브라우저가 WebKit 기반이라는 점을 고려한다.
3. Brave 전용 기능은 앱 내부 기능으로 간주하지 않는다.
   - 앱은 표준 `<audio>` 요소와 표준 웹 API만 사용한다.
   - iPhone Brave의 백그라운드 재생 기능은 사용자가 추가로 활용하는 실행 환경일 뿐이다.
4. 녹음 파일을 일반 브라우저 캐시에 저장하지 않는다.
5. 앱 데이터는 다음 구조로 저장한다.
   - 실제 오디오 파일: OPFS 우선
   - 녹음 제목·날짜·길이·재생 횟수 등 메타데이터: IndexedDB
   - OPFS를 사용할 수 없는 환경: IndexedDB에 Blob으로 저장하는 대체 방식
6. 중요한 녹음이 브라우저 데이터 삭제로 사라질 수 있으므로 파일 내보내기와 가져오기를 반드시 제공한다.
7. 마이크 기능은 HTTPS에서만 정상적으로 사용할 수 있으므로 배포 환경은 GitHub Pages, Netlify 등 HTTPS 정적 호스팅을 사용한다.
8. iPhone에서 작동한다고 추측하지 말고, 기능 감지와 대체 동작을 코드에 넣는다.
9. 지원하지 않는 기능 때문에 전체 앱이 멈추지 않도록 progressive enhancement 방식으로 구현한다.
10. 첫 버전에서는 서버, 회원가입, 클라우드 업로드, 음성 인식, 고급 편집 기능을 넣지 않는다.

## 3. 권장 기술 구성

### 기본 구성

- HTML
- CSS
- JavaScript 또는 TypeScript
- 모바일 우선 반응형 UI
- 정적 웹앱
- PWA 구성
  - `manifest.webmanifest`
  - Service Worker
  - 앱 셸 오프라인 캐시
- 실제 녹음 파일은 Service Worker 캐시에 저장하지 않는다.

프레임워크 사용 여부는 자유지만, 앱 규모가 작으므로 불필요하게 복잡한 상태관리 라이브러리나 서버 프레임워크를 도입하지 않는다.

Vanilla JavaScript로도 충분하다. 프레임워크를 사용한다면 빌드 결과가 일반 정적 파일로 배포되어야 한다.

### 사용할 주요 웹 API

- `navigator.mediaDevices.getUserMedia()`
- `MediaRecorder`
- `MediaRecorder.isTypeSupported()`
- `<audio>`
- IndexedDB
- Origin Private File System(OPFS)
- `navigator.storage.estimate()`
- `navigator.storage.persist()` 및 `persisted()`
- Media Session API
- Web App Manifest
- Service Worker
- Page Visibility API
- Wake Lock API는 지원되는 환경에서만 선택적으로 사용

## 4. 핵심 사용자 흐름

### 최초 실행

1. 앱 설명 표시
2. 브라우저 호환성 검사
3. 마이크 사용 가능 여부 검사
4. 로컬 저장공간 기능 검사
5. 저장공간 영구 보존 요청 가능 여부 검사
6. 사용자가 녹음 버튼을 누를 때 마이크 권한 요청
7. 권한이 거부되면 브라우저별 해결 방법을 간단히 안내

마이크 권한은 앱을 열자마자 자동 요청하지 말고, 반드시 사용자가 녹음 버튼을 누른 뒤 요청한다.

### 녹음

1. 사용자가 `녹음 시작`을 누른다.
2. 녹음 시간과 현재 상태를 크게 표시한다.
3. 녹음 중에는 다음 기능을 제공한다.
   - 일시정지
   - 이어서 녹음
   - 종료 및 저장
   - 취소
4. 종료 시 파일을 완성하고 로컬 저장소에 저장한다.
5. 저장이 끝나기 전까지 저장 중 상태를 표시한다.
6. 저장 성공 후 녹음 목록에 즉시 추가한다.
7. 기본 제목은 날짜와 시간을 사용한다.
   - 예: `2026-07-14 21-30 녹음`
8. 저장 후 바로 제목을 수정할 수 있게 한다.

### 재생

1. 녹음 목록에서 항목을 누르면 재생한다.
2. 하단 또는 별도 화면에 고정 플레이어를 표시한다.
3. 다음 기능을 제공한다.
   - 재생
   - 일시정지
   - 이전 녹음
   - 다음 녹음
   - 10초 뒤로
   - 10초 앞으로
   - 탐색 바
   - 현재 시간과 전체 시간
   - 셔플
   - 반복 없음
   - 한 곡 반복
   - 전체 반복
   - 재생 종료 타이머
4. 가능한 환경에서는 Media Session API로 잠금화면과 알림 영역에 다음 정보를 전달한다.
   - 제목
   - 재생/일시정지
   - 이전/다음
   - 탐색 위치
5. Media Session API가 없거나 일부 액션이 지원되지 않아도 기본 재생은 정상 작동해야 한다.

## 5. 녹음 형식 선택 규칙

파일 확장자나 MIME 타입을 하드코딩하지 않는다.

실행 시 `MediaRecorder.isTypeSupported()`를 사용하여 현재 브라우저가 실제로 지원하는 형식을 순서대로 검사한다.

권장 후보 순서의 예:

```js
const mimeCandidates = [
  'audio/mp4',
  'audio/webm;codecs=opus',
  'audio/webm',
  'audio/ogg;codecs=opus',
];

const supportedMimeType =
  mimeCandidates.find((type) => MediaRecorder.isTypeSupported(type)) ?? '';
```

중요 사항:

- Android Chromium 계열과 iPhone WebKit의 지원 형식이 다를 수 있다.
- 형식 선택 결과에 맞는 파일 확장자를 저장한다.
- 브라우저가 자동 형식을 선택하게 된 경우 `recorder.mimeType`에서 실제 형식을 확인한다.
- 재생할 때 파일명만 믿지 말고 저장된 MIME 타입을 함께 사용한다.
- iPhone 지원을 위해 WebM만을 유일한 저장 형식으로 강제하지 않는다.
- 녹음 품질 설정이 브라우저에서 무시될 수 있으므로 실제 생성된 결과를 기준으로 처리한다.

초기 녹음 품질은 지나치게 높이지 않는다. 노래 연습용으로 충분하면서 용량이 과도하지 않은 기본값을 사용한다. 지원되는 경우 `audioBitsPerSecond`를 약 96~128 kbps 범위로 요청하되, 브라우저가 이를 그대로 적용한다고 가정하지 않는다.

## 6. 녹음 중 메모리와 대용량 파일 처리

### MVP의 안정적인 기본 방식

일반적인 노래 한 곡 또는 수십 분 이내의 녹음을 우선 대상으로 한다.

- `MediaRecorder.start(timeslice)`를 사용해 일정 간격으로 Blob 조각을 받는다.
- 예: 1~5초 간격
- 녹음 종료 시 조각을 하나의 Blob으로 합친다.
- 완성된 Blob을 OPFS 또는 IndexedDB에 저장한다.
- `timeslice` 이벤트 횟수만으로 정확한 녹음 시간을 계산하지 않는다.
- 녹음 시간은 실제 시작 시각과 경과 시간을 기준으로 표시한다.

### 장시간 녹음 대응

첫 버전에서 검증되지 않은 복잡한 스트리밍 파일 병합을 무리하게 구현하지 않는다.

대신 다음 안전장치를 넣는다.

1. 녹음 시작 전 저장공간 확인
2. 녹음 중 예상 용량 표시
3. 장시간 녹음 경고
4. 저장공간이 부족할 가능성이 있으면 녹음 시작 전 안내
5. 저장 실패 시 사용자가 녹음 내용을 내보낼 수 있는 복구 경로 검토
6. 개발 구조는 향후 녹음 조각을 세션 단위로 임시 저장할 수 있게 분리

권장 1차 목표:

- 3분
- 10분
- 30분
- 60분

위 길이의 녹음을 각각 실제 기기에서 시험한다.

녹음 중 브라우저가 백그라운드로 이동하거나 화면이 잠기면 모바일 OS가 녹음을 중단하거나 JavaScript 실행을 제한할 수 있다. 따라서 녹음 화면에는 다음 안내를 표시한다.

> 안정적인 녹음을 위해 녹음 중에는 이 화면을 열어두고 화면을 잠그지 마세요.

지원되는 Android 환경에서는 Wake Lock API를 선택적으로 사용하되, iPhone에서 지원되지 않거나 거부되어도 녹음 기능이 실패하지 않아야 한다.

## 7. 저장 구조

### 권장 구조

```text
IndexedDB
├─ recordings
│  ├─ id
│  ├─ title
│  ├─ createdAt
│  ├─ duration
│  ├─ mimeType
│  ├─ fileExtension
│  ├─ size
│  ├─ storageType
│  ├─ storageKey
│  ├─ favorite
│  └─ updatedAt
├─ playlists
└─ settings

OPFS
└─ recordings
   ├─ recording-<uuid>.m4a
   ├─ recording-<uuid>.webm
   └─ ...
```

### 저장 우선순위

1. OPFS 지원 여부 검사
2. 지원되면 실제 오디오 파일은 OPFS에 저장
3. 메타데이터는 IndexedDB에 저장
4. OPFS를 사용할 수 없으면 Blob을 IndexedDB에 저장
5. 저장 방식은 각 녹음의 `storageType`에 기록
6. 재생 계층은 저장 위치와 무관하게 동일한 인터페이스를 사용

예시:

```js
class RecordingRepository {
  async save(recording) {}
  async get(id) {}
  async list() {}
  async updateMetadata(id, patch) {}
  async delete(id) {}
  async export(id) {}
  async import(file) {}
}
```

UI 코드가 OPFS나 IndexedDB를 직접 다루지 않도록 저장 계층을 분리한다.

### 저장공간 관리

설정 화면에 다음 정보를 표시한다.

- 앱이 사용 중인 저장공간
- 브라우저가 알려주는 예상 사용량
- 예상 사용 가능 용량
- 녹음 파일 개수
- 전체 녹음 용량
- 영구 저장 요청 상태
- 저장공간 새로 계산
- 파일 전체 백업 안내

가능한 경우 다음을 사용한다.

```js
await navigator.storage.estimate();
await navigator.storage.persisted();
await navigator.storage.persist();
```

`persist()`가 `false`를 반환해도 오류로 처리하지 않는다. 브라우저가 요청을 허용하지 않을 수 있음을 사용자에게 설명한다.

브라우저 저장 데이터는 영구 보관이 절대 보장되지 않는다는 안내를 설정 화면에 명확히 표시한다.

## 8. 파일 내보내기·가져오기

이 기능은 선택 기능이 아니라 필수 기능이다.

### 개별 내보내기

각 녹음 메뉴에서 다음 기능 제공:

- 파일로 저장
- 공유
- 이름 변경
- 삭제

Web Share API와 파일 공유가 지원되면 사용하고, 지원되지 않으면 일반 다운로드 방식으로 대체한다.

iPhone에서는 사용자가 파일 앱 또는 iCloud Drive로 저장할 수 있어야 한다.

### 가져오기

사용자가 스마트폰 파일 선택기를 통해 기존 오디오 파일을 앱으로 가져올 수 있게 한다.

허용할 파일 형식은 고정 확장자 목록만 믿지 말고 실제 브라우저가 `<audio>`로 재생 가능한지 검사한다.

가져온 파일도 녹음 파일과 같은 목록에서 사용할 수 있게 한다.

### 전체 백업

가능하면 다음 중 하나를 제공한다.

- 모든 녹음을 개별 파일로 순차 내보내기
- 메타데이터 JSON 내보내기
- 여러 파일을 ZIP으로 묶는 기능은 모바일 메모리 사용량을 검토한 뒤 선택적으로 구현

첫 버전에서는 대용량 ZIP 생성보다 개별 내보내기와 메타데이터 백업을 우선한다.

## 9. 재생목록·셔플·반복 규칙

### 셔플

- 원본 목록 순서를 직접 변경하지 않는다.
- 별도의 재생 큐를 생성한다.
- 현재 재생 중인 곡이 셔플 활성화 직후 갑자기 바뀌지 않게 한다.
- 셔플 중 이전 버튼을 누르면 실제 재생 이력을 따라간다.
- 같은 곡이 연속으로 불필요하게 반복되지 않게 한다.

### 반복 모드

다음 세 가지 상태를 사용한다.

```text
off
one
all
```

버튼을 누를 때마다 순환한다.

- `off`: 마지막 곡이 끝나면 정지
- `one`: 현재 곡 반복
- `all`: 목록 끝에서 처음으로 이동

### 종료 타이머

최소 다음 선택지를 제공한다.

- 사용 안 함
- 10분
- 20분
- 30분
- 60분
- 현재 곡이 끝난 뒤

남은 시간을 화면에 표시한다.

타이머가 끝나면:

1. 재생 정지
2. 타이머 상태 초기화
3. 필요하면 짧은 안내 표시

`setTimeout()` 하나만 장시간 믿지 말고 종료 예정 시각을 저장한 뒤 현재 시각과 비교한다. 모바일 브라우저가 백그라운드에서 타이머를 지연할 수 있기 때문이다.

## 10. 백그라운드 재생과 iPhone Brave 대응

### 기본 원칙

- 재생 엔진은 하나의 `<audio>` 요소를 계속 재사용한다.
- 곡마다 여러 `<audio>` 요소를 생성하지 않는다.
- Web Audio API를 기본 재생 엔진으로 사용하지 않는다.
- 일반 재생에는 표준 `<audio>` 요소를 사용한다.
- 현재 곡 종료 시 `ended` 이벤트로 다음 곡을 결정한다.
- 앱이 화면에 복귀했을 때 현재 재생 상태와 UI를 다시 동기화한다.

### Brave 관련

iPhone Brave의 Playlist 또는 백그라운드 미디어 재생은 사용자가 활용할 수 있지만, 앱의 정상 작동 조건으로 강제하지 않는다.

앱 내부 안내에는 다음 정도만 제공한다.

> iPhone에서 화면을 잠근 상태로 재생하려면 Brave의 미디어 또는 Playlist 관련 설정이 필요할 수 있습니다. 기기와 브라우저 버전에 따라 동작이 달라질 수 있습니다.

Brave 전용 내부 API나 사용자 설정을 JavaScript로 조작하려 하지 않는다.

### 반드시 확인할 항목

- 한 곡이 백그라운드에서 계속 재생되는가
- 한 곡 종료 후 다음 곡으로 넘어가는가
- 셔플 상태에서도 다음 곡으로 넘어가는가
- 한 곡 반복이 유지되는가
- 전체 반복이 유지되는가
- 잠금화면 재생/일시정지가 작동하는가
- 앱으로 돌아왔을 때 탐색 바가 올바른가
- 전화 수신, 알람, 다른 오디오 앱 실행 후 상태가 복구되는가

백그라운드에서 다음 곡 자동 전환이 특정 iPhone 환경에서 불가능하거나 불안정하면 이를 숨기지 말고 호환성 안내를 표시한다. 전면 재생은 계속 정상 작동해야 한다.

## 12. UI·UX 요구사항

- 모바일 세로 화면을 최우선으로 한다.
- 작은 아이폰 화면에서도 버튼이 겹치지 않게 한다.

- 녹음 시작과 삭제 버튼을 시각적으로 명확히 구분한다.
- 녹음 중 브라우저 뒤로가기로 실수로 페이지를 이탈하지 않게 경고한다.
- 저장 중 앱을 닫지 말라는 상태를 보여준다.
- 사용자에게 기술 용어를 과도하게 노출하지 않는다.
- 오류 메시지는 원인과 해결 방법을 함께 표시한다.
- 다크 모드와 라이트 모드를 지원한다.
- 일본어 또는 한국어 UI를 쉽게 교체할 수 있도록 문자열을 별도 객체나 파일로 관리한다.
- 첫 버전 기본 UI 언어는 개발자가 선택하되 하드코딩을 피한다.

## 13. 오류 처리

다음 상황을 각각 구분해서 처리한다.

### 마이크

- HTTPS가 아님
- 브라우저가 마이크 API를 지원하지 않음
- 사용자가 권한을 거부함
- 다른 앱이 마이크를 사용 중
- 입력 장치를 찾을 수 없음
- 녹음 중 마이크 스트림이 종료됨

### 저장

- OPFS 사용 불가
- IndexedDB 사용 불가
- 저장공간 부족
- 파일 쓰기 실패
- 메타데이터 저장 실패
- 파일은 저장됐지만 메타데이터 저장 실패
- 메타데이터는 저장됐지만 파일 저장 실패

파일과 메타데이터가 서로 어긋난 경우를 복구할 수 있도록 앱 시작 시 무결성 검사를 실행한다.

### 재생

- 파일이 없음
- 파일이 손상됨
- 현재 브라우저에서 코덱을 재생할 수 없음
- Blob URL 생성 실패
- 브라우저가 자동 재생을 차단함
- 백그라운드 전환 후 오디오 상태 불일치

재생할 파일을 바꿀 때 이전 `URL.createObjectURL()` 주소는 `URL.revokeObjectURL()`로 해제한다.

## 14. 데이터 안전 규칙

- 녹음 삭제는 확인 후 실행한다.
- 삭제 직후 잠시 실행 취소 기능을 제공하는 방안을 우선 검토한다.
- 앱 업데이트로 IndexedDB 스키마가 바뀌어도 기존 녹음을 삭제하지 않는다.
- 데이터베이스 마이그레이션 코드를 작성한다.
- Service Worker 업데이트가 실제 녹음 저장소를 삭제하거나 초기화하지 않게 한다.
- 개발 편의를 이유로 앱 시작 시 데이터베이스를 재생성하지 않는다.
- 테스트용 데이터 초기화 기능은 개발 모드에서만 사용한다.
- 로컬 데이터 삭제 전 백업 안내를 보여준다.

## 19. 완료 조건

다음 조건을 충족해야 첫 버전을 완료로 본다.

1. Android Chrome에서 녹음·저장·재생이 안정적으로 작동한다.
2. Android Brave에서도 동일한 기본 기능이 작동한다.
3. iPhone Safari에서 녹음·저장·전면 재생이 작동한다.
4. iPhone Brave에서 녹음·저장·전면 재생이 작동한다.
5. iPhone Brave 백그라운드 재생 가능 여부를 실제 기기로 확인하고 결과를 문서화한다.
6. 백그라운드 다음 곡 전환이 불안정한 경우 앱이 멈추거나 데이터를 잃지 않으며 사용자에게 제한을 알린다.
7. 녹음 파일을 앱 밖으로 내보낼 수 있다.
8. 외부 오디오 파일을 앱으로 가져올 수 있다.
9. 앱 새로고침·브라우저 종료·기기 재부팅 후에도 녹음 목록이 유지된다.
10. 저장 실패와 권한 거부가 사용자에게 이해 가능한 문장으로 표시된다.
11. 앱 업데이트로 기존 녹음이 삭제되지 않는다.
12. 지원되지 않는 API가 있어도 가능한 기능은 계속 작동한다.
13. 모든 핵심 기능이 모바일 터치 UI에서 사용 가능하다.
14. 개인정보나 녹음 파일이 기본적으로 외부 서버로 전송되지 않는다.

## 20. 에이전트 작업 규칙

1. 먼저 현재 프로젝트 파일을 확인하고 구조를 설명한다.
2. 구현 전 짧은 작업 계획을 작성한다.
3. 한 번에 전체 앱을 무리하게 완성하지 말고 Phase 1부터 구현한다.
4. 기능을 구현할 때마다 실제 실행 방법과 테스트 방법을 제공한다.
5. Android에서 성공했다는 이유로 iPhone 호환을 단정하지 않는다.
6. 브라우저 기능은 User-Agent 문자열보다 API 기능 감지를 우선한다.
7. MIME 타입, 저장 방식, 공유 방식은 기능 감지를 통해 선택한다.
8. 기존 데이터 삭제를 유발할 수 있는 변경은 하지 않는다.
9. 추후 유지보수가 가능하도록 녹음, 저장, 재생, UI 코드를 분리한다.
10. 복잡한 기술을 도입할 때는 왜 필요한지 설명하고 단순한 대안과 비교한다.
11. 불확실한 iOS 동작은 임의로 확신하지 말고 실제 테스트 항목으로 남긴다.
12. 오류를 숨기지 말고 개발자 로그와 사용자 메시지를 분리한다.
13. 완성 후 다음 문서를 작성한다.
    - `README.md`
    - 설치 및 실행 방법
    - 배포 방법
    - Android 테스트 결과
    - iPhone 테스트 결과
    - 알려진 제한사항
    - 데이터 백업 방법

## 23. 최종 목표 요약

이 프로젝트의 목표는 “Android에서만 우연히 작동하는 녹음 페이지”가 아니다.

다음 조건을 갖춘 실제 사용 가능한 모바일 웹앱을 만든다.

```text
스마트폰에서 녹음
→ 기기 내부의 웹앱 전용 저장공간에 보관
→ 녹음 목록 관리
→ 음악 플레이어처럼 셔플·반복·타이머 재생
→ 필요하면 파일 앱 또는 iCloud Drive로 백업
→ Android와 iPhone에서 가능한 범위 내 동일한 사용 경험 제공
```

iPhone 백그라운드 재생은 운영체제와 브라우저의 정책 영향을 받으므로 100% 보장을 문서로만 선언하지 않는다. 표준 API로 최대한 호환되게 구현하고, 실제 iPhone Brave와 Safari에서 검증한 결과를 최종 문서에 기록한다.
